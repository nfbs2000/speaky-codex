제9부: Course Evidence Implementation

34장: core decision point 계측 — Course Evidence는 어디에 꽂혀 있는가

온라인판은 원고 본문을 유지하면서 현재 fork의 실제 소스 링크를 함께 제공합니다.

이 장의 질문: Course Evidence는 Rust core 로직을 바꾸지 않고 어떤 결정 순간에 관측 이벤트를 남기는가?

학습 목표

이 장에서는 현재 삽입된 core instrumentation을 하나씩 읽습니다.

  • Session::build_initial_context의 skill metadata budget event를 이해한다.
  • build_compacted_history_with_limit의 compacted history selection event를 이해한다.
  • run_compact_task_inner_impl의 replacement history event를 이해한다.
  • Session::replace_compacted_history의 active history persistence event를 이해한다.
  • Course Evidence insertion이 product behavior를 바꾸지 않고 observation만 추가해야 한다는 원칙을 확인한다.

1단계: import 위치

core 쪽 insertion point는 codex_otel의 typed API를 import합니다.

use codex_otel::CourseEvidenceEvent;
use codex_otel::CourseEvidenceLevel;

이 import는 codex-rs/core/src/session/mod.rs와 codex-rs/core/src/compact.rs에서 확인할 수 있습니다.

2단계: skill metadata budget

첫 번째 대표 insertion point는 Session::build_initial_context입니다. 초기 context를 만들 때 skill metadata를 모두 넣으면 context budget을 압박할 수 있습니다. 그래서 budget을 계산하고, available skills render 결과를 보고합니다.

핵심 attributes:

attribute의미
budgetKindtoken budget인지 character budget인지
budgetLimitbudget 한도
contextWindow모델 context window
totalSkills후보 skill 개수
includedSkills포함된 skill 개수
omittedSkillsbudget 때문에 빠진 skill 개수
truncatedDescriptionChars잘린 설명 문자 수
warningEmittedwarning event가 나갔는지

이 event의 concept는 skill_metadata_budget, decision은 render_available_skills_with_budget입니다.

중요한 점은 full skill description을 기록하지 않는다는 것입니다. count와 truncation summary만 기록합니다.

3단계: compacted history selection

codex-rs/core/src/compact.rs의 build_compacted_history_with_limit는 user message를 뒤에서부터 보며 token budget 안에 들어가는 최근 메시지를 선택합니다.

Course Evidence는 이 함수에서 다음을 기록합니다.

attribute의미
maxTokenscompact replacement history에 허용된 최대 token
inputUserMessageCount입력 user message 수
consideredUserMessageCount뒤에서부터 실제 고려한 message 수
selectedUserMessageCount선택된 message 수
truncatedUserMessageCountbudget 때문에 잘린 message 수
remainingTokens선택 후 남은 token
summaryCharCountcompact summary 문자 수

이 event는 "compact가 최근 사용자 메시지를 어떤 budget 기준으로 남겼는가"를 observed로 보여 줍니다.

4단계: compaction replacement history 적용

run_compact_task_inner_impl은 compact summary를 만들고 replacement history를 조립한 뒤, 세션 history를 교체합니다.

Course Evidence는 교체 직전에 다음을 기록합니다.

attribute의미
replacementHistoryItemCount새 history item 개수
userMessageCountBeforeCompactioncompaction 전 user message 개수
initialContextInjectioninitial context를 어디에 주입했는지
referenceContextItemPersistedreference context item을 따로 persist하는지
summaryCharCountsummary 문자 수

이 event는 "compaction이 끝났고 replacement history가 실제로 만들어졌다"는 observed evidence입니다.

5단계: session compacted history state

Session::replace_compacted_history는 active session history를 교체하고 compacted rollout item을 persist합니다.

Course Evidence는 다음을 기록합니다.

attribute의미
replacementHistoryItemCountactive history로 들어간 item 수
referenceContextItemPersistedreference context persist 여부
compactedMessageCharCountcompacted summary 길이
compactedItemHasReplacementHistorycompacted item에 replacement history가 있는지

이 event는 compaction 결과가 단순 계산에서 끝난 것이 아니라 session state와 rollout persistence로 들어갔음을 보여 줍니다.

6단계: insertion 원칙

Course Evidence insertion은 다음 원칙을 지켜야 합니다.

  • core behavior를 바꾸지 않는다.
  • decision point 직후에만 작은 event를 남긴다.
  • full prompt, full tool result, private file body를 넣지 않는다.
  • event field에는 chapter/concept/evidenceLevel/sourceFunction/sourceFile을 항상 넣는다.
  • 정확히 내부 변수를 봤으면 Observed, 외부 diff면 Inferred, 실행 설정이면 Configured로 둔다.
  • Course Evidence가 없어도 product runtime은 그대로 동작해야 한다.

강좌 체크포인트

  1. compact selection event와 history replacement event를 둘 다 남기는 이유는 무엇인가?
  2. summaryCharCount는 기록해도 되고 summary full text는 기록하면 안 되는 이유는 무엇인가?
  3. Course Evidence call이 product logic 앞에 있으면 어떤 위험이 생길 수 있는가?
