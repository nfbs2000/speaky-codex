제9부: Course Evidence Implementation

33장: Rust OTel 타입 — CourseEvidenceEvent는 어떤 계약인가

온라인판은 원고 본문을 유지하면서 현재 fork의 실제 소스 링크를 함께 제공합니다.

이 장의 질문: Rust core가 교육용 내부 증거를 남길 때 어떤 typed event shape를 사용하고, OTel log/trace에는 어떤 attribute로 나가는가?

학습 목표

이 장에서는 codex-rs/otel에 추가된 Course Evidence 계약을 읽습니다.

  • CourseEvidenceLevel enum이 evidence level을 어떻게 제한하는지 확인한다.
  • CourseEvidenceEvent struct의 필수 필드를 이해한다.
  • SessionTelemetry::course_evidence()가 log와 trace event를 동시에 남기는 방식을 본다.
  • attributes JSON을 별도 payload로 둔 이유를 이해한다.

1단계: evidence level enum

codex-rs/otel/src/events/session_telemetry.rs에는 Course Evidence 전용 level enum이 있습니다.

pub enum CourseEvidenceLevel {
    Configured,
    Observed,
    Inferred,
}

as_str()는 wire value를 안정적으로 만듭니다.

Self::Configured => "configured",
Self::Observed => "observed",
Self::Inferred => "inferred",

이 enum은 임의 문자열을 막습니다. core insertion point가 event를 만들 때 evidence level을 세 가지 중 하나로 선택해야 하므로, UI와 normalizer가 안정적으로 분류할 수 있습니다.

2단계: event struct는 강좌와 코드 위치를 같이 요구한다

CourseEvidenceEvent의 필드는 다음과 같습니다.

pub struct CourseEvidenceEvent<'a> {
    pub chapter: &'a str,
    pub concept: &'a str,
    pub evidence_level: CourseEvidenceLevel,
    pub source_file: &'a str,
    pub source_function: &'a str,
    pub decision: &'a str,
    pub reason: Option<&'a str>,
    pub attributes: JsonValue,
}

여기서 chapter와 concept는 책과 연결되고, source_file과 source_function은 코드 근거와 연결됩니다. decision은 사람이 읽는 branch label이고, attributes는 실제 관측값입니다.

좋은 event는 이런 모양입니다.

session_telemetry.course_evidence(CourseEvidenceEvent {
    chapter: "part2.ch09",
    concept: "skill_metadata_budget",
    evidence_level: CourseEvidenceLevel::Observed,
    source_file: "codex-rs/core/src/session/mod.rs",
    source_function: "Session::build_initial_context",
    decision: "render_available_skills_with_budget",
    reason: Some("available skills were rendered within the thread metadata budget"),
    attributes: serde_json::json!({
        "budgetKind": "tokens",
        "budgetLimit": 2000,
        "totalSkills": 30,
        "includedSkills": 18,
        "omittedSkills": 12
    }),
});

3단계: SessionTelemetry::course_evidence()

emitter는 log_and_trace_event! macro를 사용합니다.

pub fn course_evidence(&self, event: CourseEvidenceEvent<'_>) {
    log_and_trace_event!(
        self,
        common: {
            event.name = "codex.course_evidence",
            course.chapter = %event.chapter,
            course.concept = %event.concept,
            evidence.level = %event.evidence_level.as_str(),
            source.file = %event.source_file,
            source.function = %event.source_function,
            decision = %event.decision,
            reason = event.reason,
        },
        log: {
            attributes_json = %event.attributes,
        },
        trace: {
            attributes_json = %event.attributes,
        },
    );
}

공통 attribute는 log와 trace 양쪽에 들어갑니다. attributes_json도 log와 trace 양쪽에 들어갑니다. 따라서 OTel collector가 log만 보든 trace만 보든 Course Evidence event를 잃지 않습니다.

4단계: 왜 attributes를 JSON string으로 두는가

Course Evidence의 scalar metadata는 concept마다 다릅니다.

  • skill budget: totalSkills, includedSkills, omittedSkills
  • compaction selection: maxTokens, selectedUserMessageCount, remainingTokens
  • replacement history: replacementHistoryItemCount, referenceContextItemPersisted

모든 field를 top-level OTel attribute로 고정하면 schema가 빨리 커집니다. 그래서 공통 index field는 top-level에 두고, concept-specific 값은 attributes_json에 담습니다.

이 구조의 장점:

  • 검색/필터: course.concept, evidence.level, source.file, decision
  • 세부 분석: attributes_json parse
  • 낮은 cardinality: top-level key 폭발 방지

5단계: public export

codex-rs/otel/src/lib.rs는 Course Evidence type을 public으로 re-export합니다.

pub use crate::events::session_telemetry::CourseEvidenceEvent;
pub use crate::events::session_telemetry::CourseEvidenceLevel;

core crate는 이 re-export를 통해 codex_otel::CourseEvidenceEvent와 codex_otel::CourseEvidenceLevel을 import합니다. 임의 tracing::event!를 쓰지 않고 typed emitter를 쓰는 이유는 Course Evidence 계약을 안정적으로 유지하기 위해서입니다.
