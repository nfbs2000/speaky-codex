제4부: Policy Harness & Extensibility

13장: 샌드박스와 승인 정책 — 안전성은 어떤 계층으로 서는가14장: Hooks 시스템 — 사용자는 어디서 런타임에 개입할 수 있는가15장: MCP 연결 관리자 — 외부 앱과 도구는 어떻게 붙는가

제5부: Subagents & Surfaces

16장: 리뷰 서브에이전트 — Codex는 어떻게 축소된 하니스 복제본을 만들어 검토를 외주화하는가17장: 모델 카탈로그와 새로고침 — 모델 선택은 왜 하니스 자원 관리인가18장: app-server, TUI, CLI — 같은 하니스가 어떻게 다른 표면으로 드러나는가

제6부: Harness Builder Lessons

19장: Codex가 보여 주는 하니스 엔지니어링 원칙 — 무엇을 가져가고 무엇을 버릴까

제7부: Long-Running Goal Harness

20장: /goal 강좌 로드맵 — 장시간 목표 루프를 어떻게 읽을 것인가21장: TUI 표면 — /goal 입력은 어떻게 앱 이벤트가 되는가22장: app-server와 저장 모델 — goal은 어떤 API와 row로 고정되는가23장: core goal runtime — active goal은 어떻게 다음 턴의 원인이 되는가24장: 모델 도구와 안전장치 — 모델은 goal을 어디까지 조작할 수 있는가25장: 내 하니스에 옮기기 — 장시간 목표 루프 구현 순서

제8부: Desktop SDK Runtime Evidence Harness

26장: Desktop Runtime Evidence 하니스 — Electron 앱은 실제 Codex를 어디서 실행하는가27장: 개발 실행 체인 — bridge, Vite, Electron은 어떤 순서로 뜨는가28장: TypeScript SDK 실행 어댑터 — runStreamed()는 어떻게 실제 Codex CLI가 되는가29장: Renderer와 bridge protocol — React UI는 SDK를 직접 부르지 않는다30장: chat.run에서 runStreamed()까지 — bridge server의 한 턴 실행 루프31장: Runtime Evidence artifact — raw, normalized, OTel, Trajectory는 어떻게 화면이 되는가

제9부: Course Evidence Implementation

32장: Course Evidence 로드맵 — 강좌 개념은 어떻게 실제 런타임 증거가 되는가33장: Rust OTel 타입 — CourseEvidenceEvent는 어떤 계약인가34장: core decision point 계측 — Course Evidence는 어디에 꽂혀 있는가35장: OTel export 검증 — codex.course_evidence는 log와 trace에 실제로 남는가36장: Desktop normalizer — codex.course_evidence는 어떻게 course.evidence가 되는가37장: Course Evidence UI와 다음 확장 — Runtime Evidence에서 무엇을 보고 무엇을 더 붙일 것인가
