# 37장: Course Evidence UI와 다음 확장 — Runtime Evidence에서 무엇을 보고 무엇을 더 붙일 것인가

> **이 장의 질문**: Course Evidence가 normalized event와 OTel trace에 들어간 뒤, 사용자는 Desktop Runtime Evidence에서 무엇을 확인할 수 있고 다음 구현 지점은 어디인가?

## 학습 목표

이 장에서는 Course Evidence의 사용자 표면과 확장 방향을 정리합니다.

- run summary가 Course Evidence count, level, concept, item preview를 어떻게 만드는지 본다.
- Runtime Evidence UI의 `CourseEvidenceReview`가 어떤 정보를 보여 주는지 확인한다.
- 현재 Trajectory review와 Course Evidence가 나란히 표시되는 구조를 이해한다.
- Course Evidence를 `trajectory-evaluation.json` 안에 넣으려면 어떤 파일을 확장해야 하는지 파악한다.
- live e2e에서 무엇을 증명해야 하는지 체크리스트를 만든다.

## 1단계: run summary가 Course Evidence를 요약한다

`desktop/server/evidence/run-archive.ts`의 `summarizeCourseEvidence()`는 normalized input에서 `eventType === "course.evidence"`만 모읍니다.

요약 field:

| field | 의미 |
| --- | --- |
| `count` | Course Evidence event 총개수 |
| `observedCount` | observed event 개수 |
| `inferredCount` | inferred event 개수 |
| `configuredCount` | configured event 개수 |
| `chapters` | 연결된 책 chapter 목록 |
| `concepts` | 관측된 concept 목록 |
| `items` | 최대 12개 event preview |

각 item에는 id, level, chapter, concept, decision, reason, source file/function, attributes preview, sourceEventIds가 들어갑니다.

이 summary는 `ProjectionRunSummary.courseEvidence`로 UI에 전달됩니다.

## 2단계: Course Evidence UI 카드

`desktop/src/components/v2/pages/BookVisualizationPage.tsx`의 `CourseEvidenceReview`는 `run.courseEvidence`가 없으면 렌더링하지 않습니다.

보이는 정보:

- `Course Evidence` 제목
- `Rust core decision evidence projected from codex.course_evidence` 설명
- `Claude 평가` 버튼
- observed count
- inferred count
- chapters 목록
- concepts 목록
- event item list

`Claude 평가` 버튼은 normalized artifact path를 Chat에 보내 Claude가 직접 파일을 읽고 평가하게 하는 흐름입니다. 중요한 것은 UI 카드가 점수를 자동으로 확정하지 않는다는 점입니다. 카드는 evidence를 보여 주고, judge는 명시적으로 실행해야 합니다.

## 3단계: Evidence surface 안에서 Trajectory와 나란히 둔다

현재 Evidence surface는 selected run에 대해 다음 순서로 렌더링합니다.

```tsx
<RuntimeEvidenceActions ... />
<SdkPipelineMonitor run={projectionRun} />
<EvidenceChain run={projectionRun} />
<CourseEvidenceReview run={projectionRun} />
<TrajectoryArtifactReview run={projectionRun} />
```

이 구조는 현재 구현의 진실을 반영합니다.

- Course Evidence는 run summary와 normalized/OTel artifact에 포함된다.
- Trajectory review는 tool round trip과 final answer gap을 본다.
- Course Evidence가 trajectory JSON 내부 field로 아직 복사되지는 않는다.
- 하지만 같은 selected run의 Evidence surface에서 함께 보이므로, 강사는 "코어 decision evidence"와 "prompt -> tool -> final trajectory"를 같은 맥락에서 비교할 수 있다.

## 4단계: trajectory JSON에 넣으려면 어디를 바꿔야 하나

Course Evidence summary를 `projection/trajectory-evaluation.json` 내부에도 넣고 싶다면 다음 파일을 확장해야 합니다.

| 파일 | 수정 내용 |
| --- | --- |
| `desktop/shared/evidence-protocol.ts` | `TrajectoryEvaluation`에 `courseEvidence?: CourseEvidenceSummary` 또는 더 작은 summary type 추가 |
| `desktop/server/evidence/trajectory.ts` | `input.events.filter(eventType === "course.evidence")`로 concept summary 생성 |
| `desktop/src/components/v2/pages/BookVisualizationPage.tsx` | trajectory artifact 옆의 courseEvidence section 렌더링 |
| `desktop/server/evidence/evidence.test.ts` | trajectory 생성 시 course evidence summary가 보존되는지 테스트 추가 |

주의할 점은 trajectory step의 `thought` field를 hidden reasoning처럼 쓰면 안 된다는 것입니다. Course Evidence는 별도 section이나 deterministic precheck summary로 넣어야 합니다.

## 5단계: live e2e에서 증명할 것

Course Evidence가 실제 남는지 증명하려면 다음을 확인합니다.

1. Course Evidence가 발생하는 prompt/run을 만든다.
2. run archive 아래 `raw-sdk-events.jsonl`에 `codex.course_evidence` 또는 관련 event가 있는지 확인한다.
3. `normalized-events.json`에 `eventType: "course.evidence"`가 있는지 확인한다.
4. `otel-trace.json`의 span events에 `course.chapter`, `course.concept`, `course.source.function` attribute가 있는지 확인한다.
5. `run-summary.json` 또는 loaded projection summary에 `courseEvidence.count > 0`인지 확인한다.
6. Runtime Evidence surface에 Course Evidence card가 보이는지 확인한다.
7. Claude 평가를 누르면 normalized path를 직접 읽으라는 prompt가 Chat draft/run으로 들어가는지 확인한다.

여기서 중요한 것은 prompt keyword만으로 통과시키지 않는 것입니다. 반드시 artifact file 안의 event와 sourceEventIds를 봐야 합니다.

## 6단계: 다음 insertion point 후보

현재 구현된 Course Evidence는 skill budget과 compaction 중심입니다. 다음으로 추가하기 좋은 지점은 다음과 같습니다.

| concept | 후보 파일 | 관측할 값 |
| --- | --- | --- |
| auto compaction threshold | `codex-rs/core/src/compact.rs` 또는 turn loop 주변 | context window, reserved summary token, threshold, should compact, guard reason |
| tool result truncation | `codex-rs/core/src/tools/context.rs` | original char count, truncated char count, truncation policy, source id |
| goal accounting | `codex-rs/core/src/goals.rs` | token delta, time delta, budget limit, status transition |
| permission branch | exec approval/sandbox policy 주변 | approval requirement, sandbox mode, denial/allow branch |
| retry/rate limit | model client retry path | status, retry count, backoff, final exhausted 여부 |

각 후보는 "full content"가 아니라 작은 결정 metadata만 남겨야 합니다.

## Claim -> file path -> observable event/check

| Claim | file path | observable event/check |
| --- | --- | --- |
| run summary는 Course Evidence count와 items를 만든다 | `desktop/server/evidence/run-archive.ts` | `summarizeCourseEvidence()`가 `count`, `observedCount`, `concepts`, `items`를 반환한다 |
| UI 카드는 run summary의 Course Evidence를 읽는다 | `desktop/src/components/v2/pages/BookVisualizationPage.tsx` | `CourseEvidenceReview({ run })`가 `run.courseEvidence`를 사용한다 |
| Evidence surface는 Course Evidence와 Trajectory를 함께 보여 준다 | `desktop/src/components/v2/pages/BookVisualizationPage.tsx` | `CourseEvidenceReview` 다음에 `TrajectoryArtifactReview`가 렌더링된다 |
| 현재 trajectory builder는 `course.evidence`를 step에 포함하지 않는다 | `desktop/server/evidence/trajectory.ts` | `INCLUDED_EVENTS`에 `course.evidence`가 없다 |
| trajectory 안에 Course Evidence를 넣으려면 shared type부터 확장해야 한다 | `desktop/shared/evidence-protocol.ts` | `TrajectoryEvaluation` type에 현재 `courseEvidence` field가 없다 |

## 강좌 체크포인트

1. Course Evidence card가 보인다고 해서 trajectory judge score가 자동 확정되지 않는 이유는 무엇인가?
2. live e2e에서 `normalized-events.json`과 `otel-trace.json`을 모두 확인해야 하는 이유는 무엇인가?
3. 다음 insertion point를 추가할 때 `observed`와 `inferred`를 어떻게 구분할 것인가?

이 Part의 결론은 단순합니다. Course Evidence는 책의 설명을 "있어 보이는 해설"에서 "실제 run artifact로 확인 가능한 내부 결정 증거"로 바꾸는 얇은 관측 레이어입니다. core logic은 그대로 두고, decision point마다 작은 안전한 event를 남기며, Desktop Runtime Evidence가 그 event를 raw -> normalized -> OTel -> summary -> UI로 끌어올립니다.
