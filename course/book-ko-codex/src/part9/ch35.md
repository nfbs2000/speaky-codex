# 35장: OTel export 검증 — `codex.course_evidence`는 log와 trace에 실제로 남는가

> **이 장의 질문**: Rust에서 `course_evidence()`를 호출하면 OTel log exporter와 trace exporter 양쪽에 같은 Course Evidence attribute가 남는다는 것을 어떻게 증명하는가?

## 학습 목표

이 장에서는 Course Evidence의 Rust 테스트를 읽습니다.

- in-memory log exporter와 span exporter를 붙이는 방식을 이해한다.
- `SessionTelemetry::course_evidence()` 호출 후 log attribute와 trace event attribute를 각각 검증한다.
- `codex.course_evidence`가 log-only target과 trace span event 양쪽에 routing되는 이유를 확인한다.
- 이 테스트가 Desktop UI 테스트보다 먼저 필요한 이유를 이해한다.

## 1단계: 테스트 이름이 계약이다

`codex-rs/otel/tests/suite/otel_export_routing_policy.rs`에는 다음 테스트가 있습니다.

```rust
fn otel_export_routing_policy_routes_course_evidence_log_and_trace_events()
```

이 테스트가 확인하는 계약은 분명합니다.

1. Course Evidence는 OTel log로 export된다.
2. Course Evidence는 OTel trace span event로도 export된다.
3. common attributes는 log와 trace에 동일하게 들어간다.
4. `attributes_json`도 log와 trace에 들어간다.

## 2단계: in-memory exporter로 검증한다

테스트는 `InMemoryLogExporter`와 `InMemorySpanExporter`를 만듭니다.

```rust
let log_exporter = InMemoryLogExporter::default();
let logger_provider = SdkLoggerProvider::builder()
    .with_simple_exporter(log_exporter.clone())
    .build();

let span_exporter = InMemorySpanExporter::default();
let tracer_provider = SdkTracerProvider::builder()
    .with_simple_exporter(span_exporter.clone())
    .build();
```

외부 collector나 네트워크 없이 memory 안에서 OTel output을 검사하기 때문에 빠르고 결정적입니다.

## 3단계: `SessionTelemetry`로 event를 보낸다

테스트는 실제 `SessionTelemetry`를 만들고 root span 안에서 `course_evidence()`를 호출합니다.

```rust
manager.course_evidence(CourseEvidenceEvent {
    chapter: "part6.ch19",
    concept: "compacted_history_selection",
    evidence_level: CourseEvidenceLevel::Observed,
    source_file: "codex-rs/core/src/compact.rs",
    source_function: "build_compacted_history_with_limit",
    decision: "select_recent_user_messages_until_token_budget",
    reason: Some("compact replacement history keeps recent user messages within budget"),
    attributes: json!({
        "maxTokens": 20000,
        "selectedUserMessageCount": 3,
        "selectedReplacementIds": ["msg-1", "msg-2"]
    }),
});
```

이 event는 실제 core insertion point와 같은 shape입니다. 테스트용 concept라도 chapter, source file/function, decision, attributes를 모두 채웁니다.

## 4단계: log attribute를 검증한다

테스트는 log exporter에서 `event.name = codex.course_evidence`인 log를 찾고 common attributes를 비교합니다.

검증 대상:

- `course.chapter`
- `course.concept`
- `evidence.level`
- `source.file`
- `source.function`
- `decision`
- `reason`
- `attributes_json`

특히 `attributes_json`에 `selectedReplacementIds`가 들어 있는지 확인합니다. 이것은 concept-specific payload가 손실되지 않았다는 체크입니다.

## 5단계: trace event attribute도 검증한다

그 다음 span exporter에서 finished span을 가져오고, span event 중 `codex.course_evidence`를 찾습니다.

```rust
let course_trace_event =
    find_span_event_by_name_attr(&spans[0].events.events, "codex.course_evidence");
```

log와 같은 expected common attributes를 trace event에서도 비교합니다. 이 테스트가 통과하면 Course Evidence는 log-only observability가 아니라 trace timeline에도 붙는다는 뜻입니다.

## 6단계: 왜 이 테스트가 중요한가

Desktop UI에서 Course Evidence 카드가 보이는 것은 최종 결과입니다. 하지만 그 전에 Rust OTel layer에서 event가 실제로 남는지 증명해야 합니다.

만약 이 테스트가 없다면 다음 문제가 숨어도 UI만 보고는 놓칠 수 있습니다.

- log에는 있는데 trace에는 없음
- trace에는 있는데 attributes_json이 빠짐
- event name이 달라져 normalizer가 못 찾음
- evidence level string이 바뀌어 UI count가 틀림
