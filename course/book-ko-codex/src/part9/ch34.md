# 34장: core decision point 계측 — Course Evidence는 어디에 꽂혀 있는가

> **이 장의 질문**: Course Evidence는 Rust core 로직을 바꾸지 않고 어떤 결정 순간에 관측 이벤트를 남기는가?

## 학습 목표

이 장에서는 현재 삽입된 core instrumentation을 하나씩 읽습니다.

- `Session::build_initial_context`의 skill metadata budget event를 이해한다.
- `build_compacted_history_with_limit`의 compacted history selection event를 이해한다.
- `run_compact_task_inner_impl`의 replacement history event를 이해한다.
- `Session::replace_compacted_history`의 active history persistence event를 이해한다.
- Course Evidence insertion이 product behavior를 바꾸지 않고 observation만 추가해야 한다는 원칙을 확인한다.

## 1단계: import 위치

core 쪽 insertion point는 `codex_otel`의 typed API를 import합니다.

```rust
use codex_otel::CourseEvidenceEvent;
use codex_otel::CourseEvidenceLevel;
```

이 import는 `codex-rs/core/src/session/mod.rs`와 `codex-rs/core/src/compact.rs`에서 확인할 수 있습니다.

## 2단계: skill metadata budget

첫 번째 대표 insertion point는 `Session::build_initial_context`입니다. 초기 context를 만들 때 skill metadata를 모두 넣으면 context budget을 압박할 수 있습니다. 그래서 budget을 계산하고, available skills render 결과를 보고합니다.

핵심 attributes:

| attribute | 의미 |
| --- | --- |
| `budgetKind` | token budget인지 character budget인지 |
| `budgetLimit` | budget 한도 |
| `contextWindow` | 모델 context window |
| `totalSkills` | 후보 skill 개수 |
| `includedSkills` | 포함된 skill 개수 |
| `omittedSkills` | budget 때문에 빠진 skill 개수 |
| `truncatedDescriptionChars` | 잘린 설명 문자 수 |
| `warningEmitted` | warning event가 나갔는지 |

이 event의 concept는 `skill_metadata_budget`, decision은 `render_available_skills_with_budget`입니다.

중요한 점은 full skill description을 기록하지 않는다는 것입니다. count와 truncation summary만 기록합니다.

## 3단계: compacted history selection

`codex-rs/core/src/compact.rs`의 `build_compacted_history_with_limit`는 user message를 뒤에서부터 보며 token budget 안에 들어가는 최근 메시지를 선택합니다.

Course Evidence는 이 함수에서 다음을 기록합니다.

| attribute | 의미 |
| --- | --- |
| `maxTokens` | compact replacement history에 허용된 최대 token |
| `inputUserMessageCount` | 입력 user message 수 |
| `consideredUserMessageCount` | 뒤에서부터 실제 고려한 message 수 |
| `selectedUserMessageCount` | 선택된 message 수 |
| `truncatedUserMessageCount` | budget 때문에 잘린 message 수 |
| `remainingTokens` | 선택 후 남은 token |
| `summaryCharCount` | compact summary 문자 수 |

이 event는 "compact가 최근 사용자 메시지를 어떤 budget 기준으로 남겼는가"를 observed로 보여 줍니다.

## 4단계: compaction replacement history 적용

`run_compact_task_inner_impl`은 compact summary를 만들고 replacement history를 조립한 뒤, 세션 history를 교체합니다.

Course Evidence는 교체 직전에 다음을 기록합니다.

| attribute | 의미 |
| --- | --- |
| `replacementHistoryItemCount` | 새 history item 개수 |
| `userMessageCountBeforeCompaction` | compaction 전 user message 개수 |
| `initialContextInjection` | initial context를 어디에 주입했는지 |
| `referenceContextItemPersisted` | reference context item을 따로 persist하는지 |
| `summaryCharCount` | summary 문자 수 |

이 event는 "compaction이 끝났고 replacement history가 실제로 만들어졌다"는 observed evidence입니다.

## 5단계: session compacted history state

`Session::replace_compacted_history`는 active session history를 교체하고 compacted rollout item을 persist합니다.

Course Evidence는 다음을 기록합니다.

| attribute | 의미 |
| --- | --- |
| `replacementHistoryItemCount` | active history로 들어간 item 수 |
| `referenceContextItemPersisted` | reference context persist 여부 |
| `compactedMessageCharCount` | compacted summary 길이 |
| `compactedItemHasReplacementHistory` | compacted item에 replacement history가 있는지 |

이 event는 compaction 결과가 단순 계산에서 끝난 것이 아니라 session state와 rollout persistence로 들어갔음을 보여 줍니다.

## 6단계: insertion 원칙

Course Evidence insertion은 다음 원칙을 지켜야 합니다.

- core behavior를 바꾸지 않는다.
- decision point 직후에만 작은 event를 남긴다.
- full prompt, full tool result, private file body를 넣지 않는다.
- event field에는 chapter/concept/evidenceLevel/sourceFunction/sourceFile을 항상 넣는다.
- 정확히 내부 변수를 봤으면 `Observed`, 외부 diff면 `Inferred`, 실행 설정이면 `Configured`로 둔다.
- Course Evidence가 없어도 product runtime은 그대로 동작해야 한다.

## Claim -> file path -> observable event/check

| Claim | file path | observable event/check |
| --- | --- | --- |
| skill metadata budget은 initial context assembly에서 관측된다 | `codex-rs/core/src/session/mod.rs` | `concept: "skill_metadata_budget"`와 `decision: "render_available_skills_with_budget"`가 있다 |
| compacted history selection은 token budget 결과를 기록한다 | `codex-rs/core/src/compact.rs` | `source_function: "build_compacted_history_with_limit"`와 `selectedUserMessageCount` attribute가 있다 |
| replacement history 적용은 compaction 완료 후 기록된다 | `codex-rs/core/src/compact.rs` | `decision: "apply_compacted_replacement_history"`가 있다 |
| active session history 교체도 별도 event로 남는다 | `codex-rs/core/src/session/mod.rs` | `source_function: "Session::replace_compacted_history"`가 있다 |
| 현재 event는 모두 hidden reasoning 대신 scalar metadata를 담는다 | `codex-rs/core/src/session/mod.rs`, `codex-rs/core/src/compact.rs` | attributes가 count/boolean/length/enum 중심이다 |

## 강좌 체크포인트

1. compact selection event와 history replacement event를 둘 다 남기는 이유는 무엇인가?
2. `summaryCharCount`는 기록해도 되고 summary full text는 기록하면 안 되는 이유는 무엇인가?
3. Course Evidence call이 product logic 앞에 있으면 어떤 위험이 생길 수 있는가?
