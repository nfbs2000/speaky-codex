# 24장: 모델 도구와 안전장치 — 모델은 goal을 어디까지 조작할 수 있는가

> **이 장의 질문**: active goal을 계속 수행하는 모델에게 Codex는 어떤 goal 도구를 주고, 어떤 상태 변경은 의도적으로 막는가?

## 학습 목표

이 장에서는 model-visible surface를 읽습니다.

- `get_goal`, `create_goal`, `update_goal` 세 도구의 역할을 구분한다.
- 모델이 goal을 임의로 pause/resume/budget-limit하지 못하게 막는 이유를 이해한다.
- completion accounting이 왜 `update_goal(status=complete)` 직전에 일어나는지 확인한다.
- budget limit steering과 completion report의 차이를 본다.

## 1단계: goal tool spec

`codex-rs/core/src/tools/handlers/goal_spec.rs`는 세 도구를 정의합니다.

| 도구 | 설명 |
| --- | --- |
| `get_goal` | 현재 goal, status, budget, token/time usage, remaining token budget 조회 |
| `create_goal` | 명시적으로 요청된 경우에만 새 active goal 생성 |
| `update_goal` | 기존 goal을 complete로 표시 |

중요한 문장은 `create_goal` 설명에 있습니다.

```rust
"Create a goal only when explicitly requested by the user or system/developer instructions; do not infer goals from ordinary tasks."
```

모델은 일반 작업을 보고 "이건 goal로 만들자"고 추론하면 안 됩니다. goal 생성은 사용자나 상위 instruction이 명시적으로 요청한 경우에만 허용됩니다.

## 2단계: update_goal은 complete만 허용한다

`update_goal` schema는 status enum을 하나만 노출합니다.

```rust
JsonSchema::string_enum(
    vec![json!("complete")],
    Some("Required. Set to complete only when the objective is achieved and no required work remains.".to_string()),
)
```

handler도 같은 제한을 다시 확인합니다.

```rust
if args.status != ThreadGoalStatus::Complete {
    return Err(FunctionCallError::RespondToModel(
        "update_goal can only mark the existing goal complete; pause, resume, and budget-limited status changes are controlled by the user or system"
            .to_string(),
    ));
}
```

이중 guard가 중요합니다. schema는 모델이 보는 도구 표면을 제한하고, handler는 실제 실행 시 다시 방어합니다.

## 3단계: create_goal은 기존 goal이 있으면 실패한다

`create_goal.rs`는 `session.create_thread_goal`을 호출합니다. 이미 goal이 있으면 모델에게 명확한 오류를 돌려줍니다.

```rust
"cannot create a new goal because this thread already has a goal; use update_goal only when the existing goal is complete"
```

즉 모델은 기존 goal을 임의로 교체하지 못합니다. 교체는 사용자 표면의 `/goal <objective>`와 확인 UI가 담당합니다. 모델 도구는 "새 goal 생성"과 "완료 표시" 사이에 좁은 통로만 제공합니다.

## 4단계: 완료 직전 회계

`update_goal.rs`는 complete 상태를 저장하기 전에 `GoalRuntimeEvent::ToolCompletedGoal`을 적용합니다.

```rust
session
    .goal_runtime_apply(GoalRuntimeEvent::ToolCompletedGoal {
        turn_context: turn.as_ref(),
    })
    .await?;
```

그 다음 `set_thread_goal(status=Complete)`를 호출합니다. 이 순서가 없으면 목표를 완료한 바로 그 턴의 token/time 사용량이 final usage에 빠질 수 있습니다. 특히 budgeted goal에서는 완료 보고에 token 사용량이 포함되므로, 완료 직전 accounting은 사용자에게 보이는 숫자의 정확성에 직접 연결됩니다.

## 5단계: 도구 response shape

`goal.rs`의 `GoalToolResponse`는 세 필드를 갖습니다.

```rust
struct GoalToolResponse {
    goal: Option<ThreadGoal>,
    remaining_tokens: Option<i64>,
    completion_budget_report: Option<String>,
}
```

`completion_budget_report`는 complete 상태일 때만 포함됩니다. 예를 들어 budgeted goal을 완료하면 "tokens used: X of Y"와 time used를 사용자에게 보고하라는 메시지가 tool result에 들어갑니다. 이것은 runtime이 모델에게 "완료 후 사용자에게 알려야 할 숫자"를 structured output으로 돌려주는 방식입니다.

## 6단계: budget limit은 complete가 아니다

budget limit은 모델 도구가 아니라 system/runtime이 제어합니다. state runtime의 `account_thread_goal_usage`는 token budget을 넘으면 status를 `budget_limited`로 바꿉니다. 그 뒤 core는 budget limit steering developer message를 주입할 수 있습니다.

`core/templates/goals/budget_limit.md`의 핵심 지시는 다음과 같습니다.

```markdown
The system has marked the goal as budget_limited, so do not start new substantive work for this goal. Wrap up this turn soon: summarize useful progress, identify remaining work or blockers, and leave the user with a clear next step.
```

그리고 바로 이어서 "실제로 complete가 아니면 update_goal을 호출하지 말라"고 합니다. budget exhaustion은 작업 완료가 아니라 중단 조건입니다.

## 7단계: tool exposure 자체도 feature-gated

`codex-rs/core/src/tools/spec_plan_tests.rs`는 feature가 꺼져 있을 때 goal tools가 없고, `Feature::Goals`를 켜면 `get_goal`, `create_goal`, `update_goal`이 노출되는지 검사합니다.

```rust
assert_lacks_tool_name(&tools, "get_goal");
assert_lacks_tool_name(&tools, "create_goal");
assert_lacks_tool_name(&tools, "update_goal");

features.enable(Feature::Goals);
assert_contains_tool_names(&tools, &["get_goal", "create_goal", "update_goal"]);
```

이 테스트는 model-visible surface가 runtime feature flag와 같은 게이트를 공유한다는 근거입니다.

## 안전성 패턴

| 위험 | 방어 |
| --- | --- |
| 모델이 일반 task를 goal로 승격 | `create_goal` 설명에서 명시 요청 없이는 금지 |
| 모델이 goal을 임의 교체 | `create_goal`은 기존 goal이 있으면 실패 |
| 모델이 pause/resume/budget_limit 조작 | `update_goal` enum과 handler가 `complete`만 허용 |
| budget 도달을 완료로 오인 | budget limit template이 complete와 구분 |
| final usage 누락 | `ToolCompletedGoal` accounting 후 complete 저장 |
| feature off 상태에서 도구 노출 | spec plan에서 feature flag 검사 |
