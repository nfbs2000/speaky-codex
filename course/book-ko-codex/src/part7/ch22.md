# 22장: app-server와 저장 모델 — goal은 어떤 API와 row로 고정되는가

> **이 장의 질문**: TUI가 만든 goal 이벤트는 app-server에서 어떤 protocol method가 되고, 최종적으로 어떤 sqlite row로 저장되는가?

## 학습 목표

이 장에서는 `/goal`의 control plane을 읽습니다.

- app-server protocol에서 `thread/goal/*` method를 찾는다.
- request/response/notification payload를 이해한다.
- `thread_goals` 테이블이 왜 thread당 하나의 현재 goal만 허용하는지 확인한다.
- app-server processor가 running thread와 persisted thread를 모두 처리하는 방식을 본다.

## 1단계: v2 protocol surface

`codex-rs/app-server-protocol/src/protocol/common.rs`는 app-server method vocabulary를 정의합니다. goal API는 모두 thread resource 아래에 있고 experimental marker가 붙어 있습니다.

```rust
#[experimental("thread/goal/set")]
ThreadGoalSet => "thread/goal/set" {
    params: v2::ThreadGoalSetParams,
    serialization: thread_id(params.thread_id),
    response: v2::ThreadGoalSetResponse,
},
#[experimental("thread/goal/get")]
ThreadGoalGet => "thread/goal/get" {
    params: v2::ThreadGoalGetParams,
    serialization: thread_id(params.thread_id),
    response: v2::ThreadGoalGetResponse,
},
#[experimental("thread/goal/clear")]
ThreadGoalClear => "thread/goal/clear" {
    params: v2::ThreadGoalClearParams,
    serialization: thread_id(params.thread_id),
    response: v2::ThreadGoalClearResponse,
},
```

notification도 같은 파일에 있습니다.

```rust
#[experimental("thread/goal/updated")]
ThreadGoalUpdated => "thread/goal/updated" (v2::ThreadGoalUpdatedNotification),
#[experimental("thread/goal/cleared")]
ThreadGoalCleared => "thread/goal/cleared" (v2::ThreadGoalClearedNotification),
```

## 2단계: payload shape

`codex-rs/app-server-protocol/src/protocol/v2/thread.rs`의 `ThreadGoalSetParams`는 세 가지 optional mutation을 받습니다.

```rust
pub struct ThreadGoalSetParams {
    pub thread_id: String,
    pub objective: Option<String>,
    pub status: Option<ThreadGoalStatus>,
    pub token_budget: Option<Option<i64>>,
}
```

`token_budget`이 `Option<Option<i64>>`인 이유는 "필드를 보내지 않음"과 "budget을 null로 설정"을 구분하기 위해서입니다. response는 단순합니다.

```rust
pub struct ThreadGoalSetResponse {
    pub goal: ThreadGoal,
}

pub struct ThreadGoalGetResponse {
    pub goal: Option<ThreadGoal>,
}

pub struct ThreadGoalClearResponse {
    pub cleared: bool,
}
```

`ThreadGoal`은 app-server boundary에서 camelCase로 나갑니다. 중요한 필드는 objective, status, tokenBudget, tokensUsed, timeUsedSeconds입니다.

## 3단계: sqlite schema

`codex-rs/state/migrations/0029_thread_goals.sql`은 `/goal`의 persistent shape를 한눈에 보여 줍니다.

```sql
CREATE TABLE thread_goals (
    thread_id TEXT PRIMARY KEY NOT NULL REFERENCES threads(id) ON DELETE CASCADE,
    goal_id TEXT NOT NULL,
    objective TEXT NOT NULL,
    status TEXT NOT NULL CHECK(status IN ('active', 'paused', 'budget_limited', 'complete')),
    token_budget INTEGER,
    tokens_used INTEGER NOT NULL DEFAULT 0,
    time_used_seconds INTEGER NOT NULL DEFAULT 0,
    created_at_ms INTEGER NOT NULL,
    updated_at_ms INTEGER NOT NULL
);
```

`thread_id`가 primary key입니다. 즉 한 thread에는 "현재 goal"이 하나만 있습니다. goal history를 여러 row로 쌓는 설계가 아닙니다. 대신 `goal_id`가 별도로 있어 같은 thread에서 goal이 교체되었는지 구분합니다.

## 4단계: state runtime 쿼리

`codex-rs/state/src/runtime/goals.rs`는 네 종류의 mutation을 제공합니다.

| 함수 | 의미 |
| --- | --- |
| `get_thread_goal` | thread의 현재 goal 읽기 |
| `replace_thread_goal` | 새 objective로 현재 goal 교체, usage reset |
| `update_thread_goal` | status 또는 token budget 변경 |
| `delete_thread_goal` | goal 삭제 |
| `account_thread_goal_usage` | time/token delta 누적, budget 도달 시 status 변경 |

`replace_thread_goal`은 `ON CONFLICT(thread_id) DO UPDATE`를 사용합니다. 이것이 "thread당 하나의 현재 goal"이라는 schema 설계와 맞물립니다. 새 objective가 들어오면 `goal_id`, objective, status, budget, usage counter가 모두 교체됩니다.

`update_thread_goal`과 `account_thread_goal_usage`에는 `expected_goal_id`가 들어갑니다. runtime이 어떤 goal에 대해 회계 중이었는데 사용자가 goal을 교체했다면, 이전 goal 기준 token/time delta가 새 goal에 섞이면 안 됩니다. `goal_id` filter는 이 문제를 막는 concurrency guard입니다.

## 5단계: app-server processor

`codex-rs/app-server/src/request_processors/thread_goal_processor.rs`는 protocol request를 state db mutation으로 바꿉니다. `thread_goal_set_inner`의 흐름은 다음과 같습니다.

1. `Feature::Goals` 확인
2. `thread_id` parse
3. materialized thread인지 확인하고 state db 확보
4. running thread가 있으면 rollout path 확인, 없으면 thread id로 rollout path 탐색
5. `reconcile_rollout`으로 state db thread metadata를 맞춤
6. objective와 budget 검증
7. running thread가 있으면 external mutation 전 accounting 수행
8. objective가 있으면 replace 또는 same-goal update
9. objective가 없으면 existing goal status/budget update
10. response 전송
11. ordered `ThreadGoalUpdated` notification 전송
12. running core session에 external goal runtime effect 적용

이 processor가 중요한 이유는 app-server가 단순 CRUD 계층이 아니라는 점입니다. 이미 실행 중인 core session이 있으면, persisted state 변경과 runtime accounting/scheduling 상태를 함께 맞춥니다.

## 상태 전이 표

| 요청 | state db 동작 | runtime effect |
| --- | --- | --- |
| `set(objective, Active)` | `replace_thread_goal` 또는 same objective update | active accounting mark, maybe continue |
| `set(None, Paused)` | `update_thread_goal` | active accounting clear |
| `set(None, Active)` | `update_thread_goal` | active accounting mark, maybe continue |
| `clear` | `delete_thread_goal` | stopped runtime state clear |

## Claim -> file path -> observable event/check

| Claim | file path | observable event/check |
| --- | --- | --- |
| goal API는 thread resource 아래에 있다 | `codex-rs/app-server-protocol/src/protocol/common.rs` | `thread/goal/set`, `thread/goal/get`, `thread/goal/clear` method가 있다 |
| goal update는 response와 notification을 모두 가진다 | `codex-rs/app-server-protocol/src/protocol/v2/thread.rs`, `codex-rs/app-server/src/request_processors/thread_goal_processor.rs` | `ThreadGoalSetResponse`와 `ThreadGoalUpdatedNotification`을 모두 보낸다 |
| thread당 현재 goal은 하나다 | `codex-rs/state/migrations/0029_thread_goals.sql` | `thread_id TEXT PRIMARY KEY`다 |
| goal replacement는 usage를 reset한다 | `codex-rs/state/src/runtime/goals.rs` | `replace_thread_goal`의 `ON CONFLICT` update가 `tokens_used = 0`, `time_used_seconds = 0`을 설정한다 |
| app-server는 persisted thread metadata를 reconcile한다 | `codex-rs/app-server/src/request_processors/thread_goal_processor.rs` | set/clear 전에 `reconcile_rollout`을 호출한다 |
| running thread 변경은 core runtime에도 반영된다 | `codex-rs/app-server/src/request_processors/thread_goal_processor.rs` | `apply_external_goal_set` 또는 `apply_external_goal_clear`를 호출한다 |

## 강좌 체크포인트

1. `thread_id`가 primary key인 설계는 어떤 복잡도를 줄이는가?
2. `goal_id`가 없다면 어떤 race가 생길 수 있는가?
3. app-server가 state db만 바꾸고 running core session을 건드리지 않으면 어떤 UI/runtime 불일치가 생기는가?

다음 장에서는 core runtime이 이 persisted goal을 어떻게 token/time accounting과 자동 continuation으로 바꾸는지 봅니다.
