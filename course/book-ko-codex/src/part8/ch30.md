# 30장: `chat.run`에서 `runStreamed()`까지 — bridge server의 한 턴 실행 루프

> **이 장의 질문**: 사용자가 Chat에서 프롬프트를 실행하면 bridge server는 어떤 순서로 thread, timeline, raw archive, normalized evidence를 갱신하는가?

## 학습 목표

이 장에서는 `desktop/server/index.ts`의 turn 실행 경로를 코드 레벨로 따라갑니다.

- `handleCommand()`가 `chat.run`을 `runTurn()`으로 라우팅하는 방식을 본다.
- `createThreadController()`가 SDK thread와 UI snapshot을 동시에 만드는 구조를 이해한다.
- `runTurn()`이 raw SDK event와 normalized evidence event를 같은 run archive에 저장하는 방식을 확인한다.
- `applyEvent()`와 `applyItem()`이 stream event를 Chat UI timeline/runtime entry로 바꾸는 것을 본다.

## 1단계: bridge server는 WebSocket command router다

`desktop/server/index.ts`는 WebSocket connection을 받고, message를 `ClientCommand`로 파싱해 `handleCommand()`로 넘깁니다.

```ts
socket.on("message", async (payload) => {
  const command = JSON.parse(payload.toString("utf8")) as ClientCommand;
  await handleCommand(socket, command);
});
```

`chat.run`은 `runTurn(command)`로 갑니다.

```ts
case "chat.run":
  await runTurn(command);
  return;
```

Evidence command도 같은 switch 안에 있습니다. 이 말은 Chat 실행과 Evidence 갱신이 같은 active project/session snapshot 위에서 움직인다는 뜻입니다.

## 2단계: thread controller는 SDK thread와 UI snapshot의 쌍이다

bridge server는 SDK `Thread`만 들고 있지 않습니다. UI를 위한 `ThreadSnapshot`도 같이 만듭니다.

```ts
type Controller = {
  thread: Thread;
  snapshot: ThreadSnapshot;
  turnIndex: number;
};
```

`createThreadController()`는 `codex.startThread()`를 호출하고, launch config를 snapshot에 저장합니다.

```ts
const thread = codex.startThread({
  workingDirectory: launchConfig.workingDirectory,
  skipGitRepoCheck: true,
  model: DEFAULT_MODEL ?? undefined,
  sandboxMode: DEFAULT_SANDBOX_MODE,
  approvalPolicy: DEFAULT_APPROVAL_POLICY,
  modelReasoningEffort: DEFAULT_REASONING_EFFORT,
});
```

이 launch config는 Runtime Evidence에서 configured evidence가 됩니다. 예를 들어 sandbox mode, approval policy, reasoning effort는 실제 tool call이 아니라 실행 조건입니다.

## 3단계: run archive를 먼저 만든다

`runTurn()`은 SDK stream을 돌리기 전에 run archive를 만듭니다.

```ts
const runArchive = await createRunArchive({
  rootDir: projectPath,
  runId: makeRunId(controller.snapshot.threadId, controller.turnIndex + 1),
  provider: "codex",
  threadId: controller.snapshot.threadId,
});
```

그 다음 사용자의 prompt 실행 자체를 raw와 normalized 양쪽에 남깁니다.

```ts
await runArchive.appendRaw({
  type: "client.chat.run",
  prompt: command.input,
  workingDirectory: projectPath,
  model: DEFAULT_MODEL,
  approvalPolicy: DEFAULT_APPROVAL_POLICY,
  sandboxMode: DEFAULT_SANDBOX_MODE,
});
runArchive.addNormalized(normalizeConfiguredPrompt(...));
```

여기서 prompt는 configured event입니다. 아직 tool call이나 최종 답변이 일어난 것이 아니기 때문입니다.

## 4단계: SDK event stream을 순회한다

핵심 루프는 짧지만 중요합니다.

```ts
const { events } = await controller.thread.runStreamed(command.input);
for await (const event of events) {
  await runArchive.appendRaw(event);
  applyEvent(controller, event);
  const normalizedEvents = normalizeThreadEventForEvidence({ ...event });
  runArchive.addNormalized(normalizedEvents);
  broadcastSnapshot();
}
```

이 루프에서 네 가지 일이 동시에 일어납니다.

1. raw SDK event를 JSONL archive에 append한다.
2. UI snapshot에 event를 반영한다.
3. evidence normalizer로 공통 event schema를 만든다.
4. snapshot을 모든 renderer client에게 broadcast한다.

그래서 화면의 live monitor와 디스크 artifact가 같은 source event stream에서 나옵니다.

## 5단계: `applyEvent()`는 top-level lifecycle을 처리한다

`thread.started`가 오면 local id를 실제 thread id로 교체합니다.

```ts
case "thread.started":
  if (controller.snapshot.threadId !== event.thread_id) {
    renameController(controller.snapshot.threadId, event.thread_id);
    controller.snapshot.threadId = event.thread_id;
  }
```

`turn.completed`가 오면 status를 idle로 바꾸고 usage를 저장합니다.

```ts
case "turn.completed":
  controller.snapshot.status = "idle";
  controller.snapshot.usage = normalizeUsage(event.usage);
```

이 usage는 나중에 header token count와 Runtime Evidence usage card의 근거가 됩니다.

## 6단계: `applyItem()`은 tool, file, command, assistant message를 UI event로 바꾼다

`item.completed`가 assistant message면 Chat message list에 assistant 답변을 추가합니다. command execution이면 runtime entry와 timeline event를 만듭니다. MCP tool call이면 `server.tool` 이름과 structured result/arguments preview를 남깁니다.

중요한 점은 UI runtime entry가 평가의 전부가 아니라는 것입니다. UI는 preview를 보여 주고, 실제 평가 근거는 raw/normalized artifact의 `sourceEventIds`, `toolUseId`, `tool.result`, `result.completed` 연결입니다.

## 7단계: 실패도 artifact로 남긴다

stream 중 error가 나면 bridge server는 raw에 `{ type: "error", message }`를 append하고, normalizer도 실패 evidence를 만듭니다. 그런 뒤 archive를 finalize합니다.

이 때문에 실패 run도 Runtime Evidence history에서 볼 수 있어야 합니다. 성공한 답변만 남기면 prompt 교육에서 중요한 negative evidence를 잃습니다.

## Claim -> file path -> observable event/check

| Claim | file path | observable event/check |
| --- | --- | --- |
| `chat.run`은 bridge server의 `runTurn()`으로 들어간다 | `desktop/server/index.ts` | `handleCommand()` switch에 `case "chat.run": await runTurn(command)`가 있다 |
| SDK thread와 UI snapshot은 controller 하나로 묶인다 | `desktop/server/index.ts` | `Controller` 타입이 `thread`, `snapshot`, `turnIndex`를 함께 가진다 |
| thread 생성은 SDK `startThread()` 호출이다 | `desktop/server/index.ts` | `codex.startThread({ workingDirectory, sandboxMode, approvalPolicy, ... })`가 있다 |
| raw event와 normalized event는 같은 stream loop에서 추가된다 | `desktop/server/index.ts` | `appendRaw(event)`, `normalizeThreadEventForEvidence(...)`, `addNormalized(...)`가 같은 `for await` 안에 있다 |
| turn 완료 usage는 UI snapshot에 보존된다 | `desktop/server/index.ts` | `turn.completed` 처리에서 `snapshot.usage = normalizeUsage(event.usage)`를 수행한다 |

## 강좌 체크포인트

3. 실패 run도 archive를 finalize해야 하는 이유는 무엇인가?
