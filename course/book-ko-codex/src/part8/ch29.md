# 29장: Renderer와 bridge protocol — React UI는 SDK를 직접 부르지 않는다

> **이 장의 질문**: Chat 입력 버튼을 누르면 React renderer는 어떤 command를 보내고, bridge server는 어떤 snapshot을 돌려주는가?

## 학습 목표

이 장에서는 Desktop 내부 protocol boundary를 읽습니다.

- `desktop/shared/protocol.ts`가 renderer와 bridge server의 공통 계약임을 확인한다.
- `desktop/src/api.ts`가 WebSocket transport만 담당하는 구조를 본다.
- `desktop/src/store/useCodexStore.ts`가 UI action을 command로 바꾸는 방식을 이해한다.
- renderer가 SDK 객체를 직접 들고 있지 않기 때문에 evidence 수집이 server 한 곳에 모일 수 있음을 이해한다.

## 1단계: shared protocol이 먼저다

`desktop/shared/protocol.ts`에는 renderer와 server가 공유하는 타입이 있습니다.

```ts
export type ThreadLaunchConfig = {
  source: "sdk.runStreamed";
  model: string | null;
  sandboxMode: string | null;
  approvalPolicy: string | null;
  reasoningEffort: string | null;
  workingDirectory: string;
};
```

`source: "sdk.runStreamed"`는 이 Desktop Chat이 실제 SDK streaming surface에서 왔다는 configured evidence입니다.

client command도 같은 파일에서 정의됩니다.

```ts
export type ClientCommand =
  | WorkspaceSelectCommand
  | ThreadSelectCommand
  | ThreadCreateCommand
  | ChatRunCommand
  | EvidenceRefreshCommand
  | EvidenceTrajectoryGenerateCommand
  | EvidencePromptScoreSaveCommand
  | EvidencePromptCoachGenerateCommand;
```

즉 Chat 실행, Evidence 갱신, Trajectory 생성, Prompt Score 저장이 모두 같은 WebSocket command plane을 통과합니다.

## 2단계: `api.ts`는 WebSocket transport다

`desktop/src/api.ts`는 bridge port를 읽고 WebSocket을 엽니다.

```ts
const runtimePort = import.meta.env.VITE_CODEX_VIS_PORT ?? "4318";
this.socket = new WebSocket(`ws://${hostname}:${runtimePort}`);
```

서버에서 message가 오면 JSON을 `ServerEnvelope`로 파싱해 listener에게 전달합니다.

```ts
this.socket.addEventListener("message", (event) => {
  const envelope = JSON.parse(event.data as string) as ServerEnvelope;
  for (const listener of this.listeners) {
    listener(envelope);
  }
});
```

command 전송도 단순합니다.

```ts
this.socket.send(JSON.stringify(command));
```

이 파일에는 `Codex`, `Thread`, `runStreamed`가 없습니다. renderer transport는 SDK를 모릅니다.

## 3단계: Zustand store가 UI action을 command로 바꾼다

`desktop/src/store/useCodexStore.ts`의 `boot()`는 envelope listener를 등록하고 WebSocket을 연결합니다.

```ts
api.onEnvelope((envelope) => {
  if (envelope.type === "snapshot") {
    set({ snapshot: envelope.snapshot, connection: "open" });
  } else if (envelope.type === "toast") {
    set({ toast: { level: envelope.level, message: envelope.message } });
  }
});
api.connect();
```

Chat submit은 `chat.run` command로 바뀝니다.

```ts
runTurn: (input) => {
  const activeThreadId = get().snapshot?.activeThreadId ?? undefined;
  api.send({ type: "chat.run", input, threadId: activeThreadId ?? undefined });
}
```

Evidence 버튼도 같은 방식입니다.

| UI action | command |
| --- | --- |
| 새 채팅 | `thread.create` |
| 프롬프트 실행 | `chat.run` |
| OTel 생성/갱신 | `evidence.refresh` |
| Trajectory 생성 | `evidence.trajectory.generate` |
| Prompt Score 저장 | `evidence.promptScore.save` |
| Prompt Coach 생성 | `evidence.promptCoach.generate` |

## 4단계: snapshot은 UI의 단일 읽기 모델이다

bridge server는 `ServerEnvelope` 중 `snapshot`을 방송합니다. renderer는 snapshot을 저장하고 각 페이지가 거기서 active thread, runtime entries, projection run을 읽습니다.

이 방식은 UI가 여러 파일을 직접 읽지 않아도 되게 합니다. 파일 artifact는 server가 관리하고, UI는 summary와 selected run metadata를 snapshot으로 받습니다.

## Claim -> file path -> observable event/check

| Claim | file path | observable event/check |
| --- | --- | --- |
| renderer/server protocol은 shared type으로 고정된다 | `desktop/shared/protocol.ts` | `ClientCommand`, `ServerEnvelope`, `ThreadSnapshot`이 같은 파일에 있다 |
| renderer는 SDK를 직접 import하지 않는다 | `desktop/src/api.ts`, `desktop/src/store/useCodexStore.ts` | 두 파일은 `Codex`가 아니라 `ClientCommand`와 `ServerEnvelope`만 다룬다 |
| Chat submit은 `chat.run` command가 된다 | `desktop/src/store/useCodexStore.ts` | `api.send({ type: "chat.run", input, threadId })`가 있다 |
| Evidence 조작도 같은 command plane을 쓴다 | `desktop/src/store/useCodexStore.ts` | `refreshEvidence`, `generateTrajectory`, `savePromptScore`, `generatePromptCoach`가 모두 `api.send()`를 호출한다 |
| snapshot은 UI의 읽기 모델이다 | `desktop/src/store/useCodexStore.ts` | `envelope.type === "snapshot"`일 때 `snapshot` state를 갱신한다 |

## 강좌 체크포인트

1. React component가 직접 파일 시스템을 읽지 않는 이유는 무엇인가?
2. Evidence command를 Chat command와 같은 WebSocket에 태우면 어떤 일관성이 생기는가?
3. `ThreadLaunchConfig.source`를 `"sdk.runStreamed"`로 저장하는 것은 observed인가 configured인가?

다음 장에서는 bridge server가 `chat.run`을 받아 실제 SDK thread 실행과 raw/normalized archive 생성으로 바꾸는 부분을 봅니다.
