# 27장: 개발 실행 체인 — bridge, Vite, Electron은 어떤 순서로 뜨는가

> **이 장의 질문**: `pnpm dev:electron` 류 실행에서 Desktop 앱은 어떤 포트와 환경 변수로 SDK bridge와 renderer를 묶는가?

## 학습 목표

이 장에서는 Electron 앱이 켜지는 순서를 코드로 읽습니다.

- `desktop/scripts/dev-electron.mjs`가 bridge port와 renderer port를 동적으로 잡는 방식을 본다.
- bridge server, Vite renderer, Electron shell의 시작 순서를 이해한다.
- `CODEX_VIS_PORT`, `CODEX_PROJECT_PATH`, `VITE_CODEX_VIS_PORT`, `VITE_DEV_SERVER_URL`의 역할을 분리한다.
- Electron main process가 SDK를 직접 알지 않아도 renderer가 bridge에 붙을 수 있는 이유를 확인한다.

## 1단계: free port를 두 개 잡는다

`desktop/scripts/dev-electron.mjs`는 개발 실행에서 포트를 고정하지 않습니다. `getFreePort()`로 bridge용 포트와 Vite client용 포트를 각각 잡습니다.

```js
const bridgePort = await getFreePort();
const clientPort = await getFreePort();
const devUrl = `http://${host}:${clientPort}`;
```

이 설계는 이미 다른 dev server가 떠 있어도 충돌 가능성을 줄입니다. bridge와 renderer가 서로 다른 포트를 쓰기 때문에, renderer는 compile-time env인 `VITE_CODEX_VIS_PORT`로 bridge 위치를 알아야 합니다.

## 2단계: bridge server를 먼저 띄운다

첫 번째 자식 프로세스는 bridge server입니다.

```js
spawnChild("pnpm", ["exec", "tsx", "server/index.ts"], {
  CODEX_VIS_PORT: String(bridgePort),
  CODEX_PROJECT_PATH: workspaceRoot,
});
```

여기서 중요한 환경 변수는 두 개입니다.

| env | 사용처 | 의미 |
| --- | --- | --- |
| `CODEX_VIS_PORT` | `desktop/server/index.ts` | WebSocket bridge가 listen할 포트 |
| `CODEX_PROJECT_PATH` | `desktop/server/index.ts` | 기본 workspace root, evidence artifact가 생성될 기준 경로 |

bridge server는 나중에 `ws://127.0.0.1:<CODEX_VIS_PORT>`로 renderer와 연결됩니다.

## 3단계: Vite renderer를 띄운다

두 번째 자식 프로세스는 Vite입니다.

```js
spawnChild("pnpm", ["exec", "vite", "--host", host, "--port", String(clientPort), "--strictPort"], {
  VITE_CODEX_VIS_PORT: String(bridgePort),
});
```

`VITE_CODEX_VIS_PORT`는 renderer bundle에서 `import.meta.env.VITE_CODEX_VIS_PORT`로 읽힙니다. 즉 renderer는 Electron IPC가 아니라 WebSocket으로 bridge에 붙습니다.

## 4단계: renderer가 준비된 뒤 Electron을 띄운다

스크립트는 `waitForHttp(devUrl)`로 Vite가 응답할 때까지 기다린 뒤 Electron을 실행합니다.

```js
spawnChild("pnpm", ["exec", "electron", electronMain], {
  VITE_DEV_SERVER_URL: devUrl,
  CODEX_VIS_PORT: String(bridgePort),
});
```

Electron main process는 `VITE_DEV_SERVER_URL`만 보면 됩니다. 실제 Codex bridge port는 renderer가 이미 Vite env로 알고 있습니다.

## 5단계: Electron main process는 창만 만든다

`desktop/electron/main.cjs`는 `BrowserWindow`를 만들고 dev URL 또는 build된 `dist/index.html`을 로드합니다.

```js
if (devServerUrl) {
  void window.loadURL(devServerUrl);
} else {
  const indexUrl = pathToFileURL(path.join(__dirname, "..", "dist", "index.html")).toString();
  void window.loadURL(indexUrl);
}
```

여기에는 `Codex`, `runStreamed`, `raw-sdk-events.jsonl` 같은 개념이 없습니다. 이 분리가 유지되어야 Electron shell이 단순하고, SDK 실행/증거 저장은 bridge server에서 독립적으로 검증할 수 있습니다.

## 실행 체인 요약

```text
dev-electron.mjs
  1. bridgePort 확보
  2. clientPort 확보
  3. CODEX_VIS_PORT=bridgePort pnpm exec tsx server/index.ts
  4. VITE_CODEX_VIS_PORT=bridgePort pnpm exec vite --port clientPort
  5. VITE_DEV_SERVER_URL=http://127.0.0.1:clientPort pnpm exec electron electron/main.cjs
```

## Claim -> file path -> observable event/check

| Claim | file path | observable event/check |
| --- | --- | --- |
| bridge와 renderer는 다른 포트를 쓴다 | `desktop/scripts/dev-electron.mjs` | `bridgePort`와 `clientPort`를 각각 `getFreePort()`로 만든다 |
| bridge server는 workspace root를 env로 받는다 | `desktop/scripts/dev-electron.mjs`, `desktop/server/index.ts` | `CODEX_PROJECT_PATH`가 `DEFAULT_PROJECT_PATH`가 된다 |
| renderer는 bridge port를 Vite env로 받는다 | `desktop/scripts/dev-electron.mjs`, `desktop/src/api.ts` | `VITE_CODEX_VIS_PORT`가 WebSocket URL에 들어간다 |
| Electron은 SDK를 직접 실행하지 않는다 | `desktop/electron/main.cjs` | `loadURL()` 또는 `loadURL(indexUrl)`만 수행한다 |
| child process가 비정상 종료되면 전체 dev session도 종료된다 | `desktop/scripts/dev-electron.mjs` | `bridge.on("exit")`, `vite.on("exit")`, `electron.on("exit")`가 `shutdown()`으로 연결된다 |

## 강좌 체크포인트
