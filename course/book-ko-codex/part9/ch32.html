제9부: Course Evidence Implementation

32장: Course Evidence 로드맵 — 강좌 개념은 어떻게 실제 런타임 증거가 되는가

온라인판은 원고 본문을 유지하면서 현재 fork의 실제 소스 링크를 함께 제공합니다.

이 장의 질문: 책에서 설명한 "내부 동작의 섬세함"을 프롬프트 키워드나 tool name이 아니라 실제 코드 실행 증거로 만들려면 어떤 구조가 필요한가?

학습 목표

이 장에서는 Course Evidence의 목적과 구현 범위를 잡습니다.

  • Course Evidence가 hidden chain-of-thought가 아니라 내부 변수/분기 관측 이벤트임을 이해한다.
  • observed, configured, inferred evidence level을 분리한다.
  • Rust core decision point에서 codex.course_evidence OTel event를 만들고, Desktop Runtime Evidence로 투영하는 전체 파이프라인을 본다.
  • 현재 구현된 범위와 아직 trajectory JSON에 직접 넣지 않은 경계를 명확히 구분한다.

1단계: Course Evidence가 해결하는 문제

일반 SDK event만 보면 다음은 어느 정도 보입니다.

  • 어떤 turn이 시작됐는가
  • 어떤 tool/item이 실행됐는가
  • 최종 답변과 token usage가 무엇인가
  • 실패 event가 있었는가

하지만 책에서 말하는 내부 결정은 일반 tool trace만으로는 부족합니다.

책의 개념tool trace만으로 가능한가필요한 Course Evidence
skill metadata budget일부 추정 가능total/included/omitted/truncated count
compacted history selection결과 diff로 추정 가능considered/selected/truncated user message count
replacement history 적용외부에서 보기 어려움replacement history item count, reference context persist 여부
compaction state persistenceraw event만으로 불완전compacted item과 active session history 전환 event

Course Evidence는 이 결정을 코드 안에서 작은 scalar metadata로 관측합니다.

2단계: 관측하는 것과 관측하지 않는 것

Course Evidence는 "모델의 생각"을 기록하지 않습니다. 코드가 이미 계산한 상태와 branch decision만 기록합니다.

관측하는 것:

  • count
  • boolean
  • threshold
  • enum branch
  • source file/function
  • decision label
  • short reason
  • redacted id 또는 hash

관측하지 않는 것:

  • hidden chain-of-thought
  • full prompt
  • full tool result
  • secret/env token
  • private file body
  • 모델 내부 reasoning

이 경계가 무너지면 교육용 관측성이 아니라 private data logging이 됩니다.

3단계: evidence level

Course Evidence에서는 evidence level을 명확히 나눕니다.

level의미예시
observedcore/adapter가 실제 내부 변수나 branch를 직접 기록함selectedUserMessageCount, omittedSkills, referenceContextItemPersisted
configured실행 설정이나 정책으로 주어진 값model, sandbox mode, approval policy
inferred외부 event 전후 diff로 재구성한 판단tool_result 전후 크기 비교로 budget pressure 추정

현재 Rust Course Evidence type에는 Configured, Observed, Inferred가 있습니다. not_proven은 UI/강좌 판단에서 쓰는 상태이지 Rust enum에는 아직 없습니다. 어떤 run에서 Course Evidence event가 없으면 그 개념은 "이 run에서 증명되지 않음"으로 다뤄야 합니다.

4단계: 전체 파이프라인

Rust core decision point
  -> SessionTelemetry::course_evidence(CourseEvidenceEvent)
  -> OTel log + trace event: codex.course_evidence
  -> TypeScript SDK/CLI event stream 또는 OTel export
  -> desktop normalizer: course.evidence
  -> normalized-events.json
  -> otel-trace.json event attributes
  -> run-summary.json courseEvidence
  -> Runtime Evidence UI Course Evidence card

이 파이프라인의 핵심은 "책 개념"과 "소스 코드 decision point"가 함께 저장된다는 것입니다. event 하나에는 chapter, concept, source file/function, decision, reason, attributes가 들어갑니다.

5단계: 현재 구현 범위

현재 구현은 다음까지 되어 있습니다.

  • Rust OTel type: CourseEvidenceEvent, CourseEvidenceLevel
  • Rust emitter: SessionTelemetry::course_evidence(...)
  • core insertion point: skill metadata budget, compaction history selection, compaction replacement history, session compacted history state
  • Rust OTel routing test: log와 trace 양쪽에 codex.course_evidence가 남는지 검증
  • desktop normalizer: codex.course_evidence를 course.evidence normalized event로 변환
  • desktop OTel trace: course attributes를 span event attributes로 투영
  • run summary: concept/chapter/count/items 요약
  • UI: Runtime Evidence surface에서 Course Evidence review 카드 표시

현재 구현에서 주의할 점도 있습니다.

  • trajectory-evaluation.json 자체에는 아직 courseEvidence summary를 복사하지 않습니다.
  • 대신 selected run의 courseEvidence summary를 Trajectory review와 같은 Runtime Evidence surface에서 나란히 보여 줍니다.
  • Trajectory JSON 내부로 넣으려면 desktop/shared/evidence-protocol.ts와 desktop/server/evidence/trajectory.ts를 확장해야 합니다.

강좌 체크포인트

  1. "프롬프트에 compaction이라고 썼다"가 Course Evidence가 아닌 이유는 무엇인가?
  2. 내부 변수 관측과 hidden chain-of-thought 수집의 차이는 무엇인가?
