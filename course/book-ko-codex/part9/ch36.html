제9부: Course Evidence Implementation

36장: Desktop normalizer — codex.course_evidence는 어떻게 course.evidence가 되는가

온라인판은 원고 본문을 유지하면서 현재 fork의 실제 소스 링크를 함께 제공합니다.

이 장의 질문: Rust OTel event가 Desktop Runtime Evidence에서 쓸 수 있는 normalized event로 바뀌려면 어떤 field mapping이 필요한가?

학습 목표

이 장에서는 Desktop evidence normalizer와 OTel trace builder를 읽습니다.

  • normalizeThreadEventForEvidence()가 Course Evidence event를 감지하는 조건을 확인한다.
  • normalizeCourseEvidence()가 OTel attribute와 raw event shape를 모두 받아들이는 방식을 이해한다.
  • normalized event의 courseChapter, courseConcept, courseDecision, courseAttributesJson field를 읽는다.
  • buildOtelTrace()가 normalized course.evidence를 다시 OTel span event attribute로 투영하는 구조를 본다.

1단계: normalizer의 감지 조건

desktop/server/evidence/codex-event-normalizer.ts는 event type이나 event name이 codex.course_evidence이면 Course Evidence normalizer로 보냅니다.

const eventType = asString(input.event.type);
const eventName = asString(input.event["event.name"]) || asString(input.event.name);
if (eventType === "codex.course_evidence" || eventName === "codex.course_evidence") {
  return normalizeCourseEvidence(input, input.event);
}

이 조건 때문에 source가 raw SDK event shape든 OTel log-like shape든 같은 normalized event로 바꿀 수 있습니다.

2단계: attribute alias를 읽는다

normalizeCourseEvidence()는 field name이 조금 달라도 읽을 수 있게 alias를 둡니다.

const chapter = asString(attrValue(attrs, "course.chapter", "chapter"));
const concept = asString(attrValue(attrs, "course.concept", "concept"));
const decision = asString(attrValue(attrs, "decision", "course.decision"));
const sourceFile = asString(attrValue(attrs, "source.file", "course.source.file", "sourceFile"));
const sourceFunction = asString(attrValue(attrs, "source.function", "course.source.function", "sourceFunction"));
const level = asString(attrValue(attrs, "evidence.level", "evidenceLevel"));

이것은 adapter/OTel exporter 차이 때문에 field prefix가 약간 달라질 수 있는 현실을 흡수합니다.

3단계: source id는 stable hash로 만든다

Course Evidence는 같은 concept가 여러 번 나올 수 있습니다. normalizer는 chapter, concept, decision, source, attributes를 stable stringify/hash해서 source id를 만듭니다.

const sourceId = `${input.threadId}:course_evidence:${stableHash(stableStringify([
  chapter,
  concept,
  decision,
  sourceFile,
  sourceFunction,
  attributes,
]))}`;

이 id는 normalized event의 sourceEventIds에 들어갑니다. 나중에 Claude 평가나 UI에서 "이 판단은 어떤 source event에서 왔는가"를 추적할 수 있습니다.

4단계: normalized event shape

Course Evidence는 normalized event에서 다음 field로 보존됩니다.

{
  eventType: "course.evidence",
  actor: "runtime",
  level: level === "configured" || level === "inferred" ? level : "observed",
  inputFocus: [chapter, concept, decision].filter(Boolean).join(" / "),
  resultPreview: snippet(reason || attributesJson),
  evaluation: [decision, reason].filter(Boolean).join(": "),
  courseChapter: chapter,
  courseConcept: concept,
  courseDecision: decision,
  courseReason: reason,
  courseSourceFile: sourceFile,
  courseSourceFunction: sourceFunction,
  courseAttributesJson: attributesJson,
}

level fallback이 중요합니다. 이상한 값이 들어오면 observed로 둡니다. 하지만 구현에서는 Rust enum을 쓰므로 정상 경로에서는 configured, observed, inferred 중 하나가 들어옵니다.

5단계: Desktop OTel trace로 다시 투영한다

desktop/server/evidence/otel-trace.ts는 normalized events를 OTel-compatible JSON trace로 만듭니다. Course Evidence event에는 course attribute가 붙습니다.

attr("course.chapter", event.courseChapter || ""),
attr("course.concept", event.courseConcept || ""),
attr("course.decision", event.courseDecision || ""),
attr("course.reason", event.courseReason || ""),
attr("course.source.file", event.courseSourceFile || ""),
attr("course.source.function", event.courseSourceFunction || ""),
attr("course.attributes_json", event.courseAttributesJson || "", ...),

즉 Rust OTel event가 Desktop normalized event가 되고, 다시 Desktop artifact용 OTel trace event가 됩니다. 이중 투영처럼 보이지만 역할이 다릅니다.

  • Rust OTel: core runtime에서 나온 원 관측
  • Desktop normalized: 평가/trajectory/prompt score가 읽을 공통 schema
  • Desktop OTel trace: artifact history와 외부 observability 도구가 읽을 trace 문서

6단계: desktop normalizer 테스트

desktop/server/evidence/evidence.test.ts에는 Course Evidence normalizer 테스트가 있습니다.

이 테스트는 event.name: "codex.course_evidence" event를 넣고 다음을 확인합니다.

  • normalized event type이 course.evidence인지
  • level이 observed인지
  • chapter/concept/sourceFunction이 보존되는지
  • attributes JSON에 selectedUserMessageCount가 남는지
  • OTel trace event attribute에 course.chapter, course.concept, course.source.function이 들어가는지

이 테스트는 Rust OTel 테스트와 짝을 이룹니다. Rust 쪽은 "event가 나온다"를 증명하고, Desktop 쪽은 "event를 Runtime Evidence schema로 읽는다"를 증명합니다.
