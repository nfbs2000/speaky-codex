제9부: Course Evidence Implementation

37장: Course Evidence UI와 다음 확장 — Runtime Evidence에서 무엇을 보고 무엇을 더 붙일 것인가

온라인판은 원고 본문을 유지하면서 현재 fork의 실제 소스 링크를 함께 제공합니다.

이 장의 질문: Course Evidence가 normalized event와 OTel trace에 들어간 뒤, 사용자는 Desktop Runtime Evidence에서 무엇을 확인할 수 있고 다음 구현 지점은 어디인가?

학습 목표

이 장에서는 Course Evidence의 사용자 표면과 확장 방향을 정리합니다.

  • run summary가 Course Evidence count, level, concept, item preview를 어떻게 만드는지 본다.
  • Runtime Evidence UI의 CourseEvidenceReview가 어떤 정보를 보여 주는지 확인한다.
  • 현재 Trajectory review와 Course Evidence가 나란히 표시되는 구조를 이해한다.
  • Course Evidence를 trajectory-evaluation.json 안에 넣으려면 어떤 파일을 확장해야 하는지 파악한다.
  • live e2e에서 무엇을 증명해야 하는지 체크리스트를 만든다.

1단계: run summary가 Course Evidence를 요약한다

desktop/server/evidence/run-archive.ts의 summarizeCourseEvidence()는 normalized input에서 eventType === "course.evidence"만 모읍니다.

요약 field:

field의미
countCourse Evidence event 총개수
observedCountobserved event 개수
inferredCountinferred event 개수
configuredCountconfigured event 개수
chapters연결된 책 chapter 목록
concepts관측된 concept 목록
items최대 12개 event preview

각 item에는 id, level, chapter, concept, decision, reason, source file/function, attributes preview, sourceEventIds가 들어갑니다.

이 summary는 ProjectionRunSummary.courseEvidence로 UI에 전달됩니다.

2단계: Course Evidence UI 카드

desktop/src/components/v2/pages/BookVisualizationPage.tsx의 CourseEvidenceReview는 run.courseEvidence가 없으면 렌더링하지 않습니다.

보이는 정보:

  • Course Evidence 제목
  • Rust core decision evidence projected from codex.course_evidence 설명
  • Claude 평가 버튼
  • observed count
  • inferred count
  • chapters 목록
  • concepts 목록
  • event item list

Claude 평가 버튼은 normalized artifact path를 Chat에 보내 Claude가 직접 파일을 읽고 평가하게 하는 흐름입니다. 중요한 것은 UI 카드가 점수를 자동으로 확정하지 않는다는 점입니다. 카드는 evidence를 보여 주고, judge는 명시적으로 실행해야 합니다.

3단계: Evidence surface 안에서 Trajectory와 나란히 둔다

현재 Evidence surface는 selected run에 대해 다음 순서로 렌더링합니다.

<RuntimeEvidenceActions ... />
<SdkPipelineMonitor run={projectionRun} />
<EvidenceChain run={projectionRun} />
<CourseEvidenceReview run={projectionRun} />
<TrajectoryArtifactReview run={projectionRun} />

이 구조는 현재 구현의 진실을 반영합니다.

  • Course Evidence는 run summary와 normalized/OTel artifact에 포함된다.
  • Trajectory review는 tool round trip과 final answer gap을 본다.
  • Course Evidence가 trajectory JSON 내부 field로 아직 복사되지는 않는다.
  • 하지만 같은 selected run의 Evidence surface에서 함께 보이므로, 강사는 "코어 decision evidence"와 "prompt -> tool -> final trajectory"를 같은 맥락에서 비교할 수 있다.

4단계: trajectory JSON에 넣으려면 어디를 바꿔야 하나

Course Evidence summary를 projection/trajectory-evaluation.json 내부에도 넣고 싶다면 다음 파일을 확장해야 합니다.

파일수정 내용
desktop/shared/evidence-protocol.tsTrajectoryEvaluation에 courseEvidence?: CourseEvidenceSummary 또는 더 작은 summary type 추가
desktop/server/evidence/trajectory.tsinput.events.filter(eventType === "course.evidence")로 concept summary 생성
desktop/src/components/v2/pages/BookVisualizationPage.tsxtrajectory artifact 옆의 courseEvidence section 렌더링
desktop/server/evidence/evidence.test.tstrajectory 생성 시 course evidence summary가 보존되는지 테스트 추가

주의할 점은 trajectory step의 thought field를 hidden reasoning처럼 쓰면 안 된다는 것입니다. Course Evidence는 별도 section이나 deterministic precheck summary로 넣어야 합니다.

5단계: live e2e에서 증명할 것

Course Evidence가 실제 남는지 증명하려면 다음을 확인합니다.

  1. Course Evidence가 발생하는 prompt/run을 만든다.
  2. run archive 아래 raw-sdk-events.jsonl에 codex.course_evidence 또는 관련 event가 있는지 확인한다.
  3. normalized-events.json에 eventType: "course.evidence"가 있는지 확인한다.
  4. otel-trace.json의 span events에 course.chapter, course.concept, course.source.function attribute가 있는지 확인한다.
  5. run-summary.json 또는 loaded projection summary에 courseEvidence.count > 0인지 확인한다.
  6. Runtime Evidence surface에 Course Evidence card가 보이는지 확인한다.
  7. Claude 평가를 누르면 normalized path를 직접 읽으라는 prompt가 Chat draft/run으로 들어가는지 확인한다.

여기서 중요한 것은 prompt keyword만으로 통과시키지 않는 것입니다. 반드시 artifact file 안의 event와 sourceEventIds를 봐야 합니다.

강좌 체크포인트

  1. Course Evidence card가 보인다고 해서 trajectory judge score가 자동 확정되지 않는 이유는 무엇인가?
  2. live e2e에서 normalized-events.json과 otel-trace.json을 모두 확인해야 하는 이유는 무엇인가?
  3. 다음 insertion point를 추가할 때 observed와 inferred를 어떻게 구분할 것인가?

이 Part의 결론은 단순합니다. Course Evidence는 책의 설명을 "있어 보이는 해설"에서 "실제 run artifact로 확인 가능한 내부 결정 증거"로 바꾸는 얇은 관측 레이어입니다. core logic은 그대로 두고, decision point마다 작은 안전한 event를 남기며, Desktop Runtime Evidence가 그 event를 raw -> normalized -> OTel -> summary -> UI로 끌어올립니다.
