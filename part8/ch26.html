제8부: Desktop SDK Runtime Evidence Harness

26장: Desktop Runtime Evidence 하니스 — Electron 앱은 실제 Codex를 어디서 실행하는가

온라인판은 원고 본문을 유지하면서 현재 fork의 실제 소스 링크를 함께 제공합니다.

이 장의 질문: desktop Electron 앱은 단순 UI인가, 아니면 실제 Codex SDK 실행과 Runtime Evidence 수집을 연결하는 하니스인가?

학습 목표

이 장에서는 Desktop 앱을 "책 시각화 앱"이 아니라 "Chat-first SDK runtime harness"로 읽습니다.

  • Electron, Vite renderer, bridge server의 책임을 분리한다.
  • 실제 Codex 실행 지점이 Electron main process가 아니라 desktop/server/index.ts임을 확인한다.
  • Book은 진입점이고, Runtime Evidence가 증명 surface라는 구조를 이해한다.
  • 한 번의 Chat 실행이 SQLite raw_events와 raw/normalized export artifact를 만들고, 명시 OTel 생성/갱신이 DB에서 trace artifact를 만드는 전체 흐름을 잡는다.

1단계: 세 프로세스 구조

Desktop 개발 실행은 세 프로세스를 띄웁니다.

프로세스파일책임
bridge serverdesktop/server/index.tsTypeScript SDK로 Codex thread를 만들고 event stream을 수집한다
rendererdesktop/src/*React UI, Chat, Book, Evidence surface를 렌더링한다
Electron shelldesktop/electron/main.cjsBrowserWindow를 만들고 renderer URL 또는 build artifact를 로드한다

중요한 점은 Electron main process가 Codex SDK를 직접 실행하지 않는다는 것입니다. Electron은 창을 열 뿐이고, 실제 SDK 실행과 evidence archive는 bridge server가 담당합니다.

User
  -> React ChatInput
  -> useCodexStore.runTurn()
  -> WebSocket command: chat.run
  -> desktop/server/index.ts
  -> sdk/typescript Codex.startThread().runStreamed()
  -> codex exec --experimental-json
  -> ThreadEvent stream
  -> SQLite raw_events append
  -> raw-sdk-events.jsonl / normalized-events.json export
  -> Evidence 버튼: DB raw_events -> raw / normalized / otel 재export
  -> WebSocket snapshot
  -> Chat + Runtime Evidence UI

2단계: Book과 Evidence의 역할 분리

현재 UX에서 Book은 런타임 증거를 무한 목록으로 보여 주는 곳이 아닙니다.

  • Book: 강좌 문맥, 장 내용, Try Chat 진입점
  • Chat: 실제 프롬프트 실행 surface
  • Evidence: 현재 run의 raw, normalized, OTel, trajectory, prompt score, Course Evidence를 확인하는 증명 surface

이 구조가 중요한 이유는 강좌 문장을 근거로 판단하지 않기 위해서입니다. "책에 token budget이라고 쓰여 있다"는 증거가 아닙니다. 실제 run에서 SDK event 또는 Course Evidence event가 남아야 Runtime Evidence로 인정합니다.

3단계: Desktop 하니스가 수집하는 최소 단위

Desktop bridge server는 한 turn을 실행할 때 먼저 SQLite raw_events에 SDK event를 저장합니다. 동시에 run artifact로 raw와 normalized 층을 남겨 즉시 UI가 볼 수 있게 합니다. OTel trace는 Runtime Evidence의 OTel 생성 또는 OTel 갱신 버튼을 눌렀을 때 SQLite raw_events를 우선 읽어 raw / normalized / OTel artifact를 export합니다. DB 이벤트가 없는 과거 run만 기존 normalized artifact fallback을 사용합니다.

층산출물의미
canonical raw storeobservability/data/codex-runtime.db의 raw_events현재 Chat session의 원천 런타임 이벤트 저장소
rawobservability/data/<run-id>/raw-sdk-events.jsonlSDK/CLI에서 나온 원 이벤트 보존
normalizedobservability/data/<run-id>/normalized-events.jsonprompt/tool/result/claim/evaluation/course evidence를 공통 schema로 정규화
OTelobservability/data/<run-id>/otel-trace.json명시 갱신 시 DB raw_events를 OTel-compatible trace로 투영

SQLite raw_events가 canonical runtime event store이고, JSONL/JSON artifact는 재평가 가능한 export 결과물입니다. OTel 생성/갱신으로 trace까지 만들어지면 Genkit, Opik, trajectory가 없어도 partial success입니다.
