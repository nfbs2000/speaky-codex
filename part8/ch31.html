제8부: Desktop SDK Runtime Evidence Harness

31장: Runtime Evidence artifact — raw, normalized, OTel, Trajectory는 어떻게 화면이 되는가

온라인판은 원고 본문을 유지하면서 현재 fork의 실제 소스 링크를 함께 제공합니다.

이 장의 질문: 한 번의 Chat 실행으로 생성된 SDK event는 어떤 파일로 저장되고, Desktop UI에서는 어떤 Evidence surface로 보이는가?

학습 목표

이 장에서는 run archive와 Evidence UI를 연결합니다.

  • SQLite raw_events가 current session의 canonical runtime store가 되는 이유를 본다.
  • raw archive, normalized events, OTel trace, run summary가 DB에서 어떻게 export되는지 이해한다.
  • BookVisualizationPage의 현재 역할을 "Book Reader surface"와 "Runtime Evidence surface"로 분리해 읽는다.
  • Trajectory, Prompt Score, Prompt Coach가 자동 실행이 아니라 명시 command로 생성되는 이유를 확인한다.

1단계: run directory는 observability/data/<run-id>다

desktop/server/evidence/run-archive.ts는 run root를 이렇게 정합니다.

function runRoot(rootDir: string) {
  return path.join(rootDir, "observability", "data");
}

createRunArchive()는 run마다 다음 파일을 준비합니다.

const rawJsonlPath = path.join(runDir, "raw-sdk-events.jsonl");
const normalizedPath = path.join(runDir, "normalized-events.json");
const otelPath = path.join(runDir, "otel-trace.json");
const projectionDir = path.join(runDir, "projection");

이 경로가 중요한 이유는 UI와 Claude 평가 prompt가 "어떤 파일을 직접 읽을 것인지" 명확히 알 수 있기 때문입니다.

2단계: SQLite raw_events가 원천 저장소다

Chat 실행 중 SDK event가 들어오면 desktop/server/index.ts는 먼저 SQLite에 raw event를 저장합니다.

desktop/server/index.ts
  -> persistRuntimeRawEvent()
  -> RuntimeEventStore.persistRawEvent()
  -> observability/data/codex-runtime.db
  -> raw_events(session_id, event_type, payload, ts)

이 DB가 current session의 canonical runtime event store입니다. raw-sdk-events.jsonl은 원천이 아니라 export artifact입니다.

3단계: raw archive는 sanitize 후 append/export된다

appendRaw()와 SQLite store는 원 event를 그대로 쓰기 전에 같은 sanitize 규칙을 통과시킵니다.

appendRaw: async (value) => {
  await appendFile(rawJsonlPath, `${JSON.stringify(sanitizeRawArchiveValue(value))}\n`, "utf8");
}

secret, token, private path, 과도한 private text를 그대로 남기지 않기 위한 방어선입니다. Runtime Evidence는 "증거"를 남기지만, 증거 수집이 private data dump가 되면 안 됩니다.

4단계: finalize에서는 normalized가 먼저 생성된다

finalize()는 메모리에 모은 normalized events를 normalized-events.json으로 씁니다. 이 시점에는 otel-trace.json을 자동 생성하지 않습니다.

const inputDocument: ProjectionInput = {
  runId,
  provider: input.provider ?? "codex",
  events,
};
await writeJson(normalizedPath, inputDocument);

그 뒤 run-summary.json을 씁니다. 이 summary에는 raw/normalized/OTel 존재 여부, trajectory/prompt score/prompt coach 생성 여부, Opik parity 결과, Course Evidence summary 같은 UI용 metadata가 붙습니다. Chat run 직후에는 raw와 normalized는 ready이고, OTel은 missing일 수 있습니다.

5단계: Evidence refresh는 DB에서 raw/normalized/OTel을 재export한다

desktop/server/index.ts의 handleEvidenceRefresh()는 selected run의 runId를 SQLite raw_events.session_id로 먼저 조회합니다.

const dbExport = runtimeEventStore
  ? await exportRunArtifactsFromRawEvents({
      rootDir: projectPath,
      runId: run.runId,
      eventStore: runtimeEventStore,
      provider: "codex",
    })
  : null;
latestProjectionRun = dbExport ?? await refreshOtelTrace(projectPath, run.runId) ?? run;

이 단계는 Chat 실행과 분리되어 있습니다. Chat만 실행해도 SQLite raw_events와 raw/normalized export는 남지만, OTel trace는 사용자가 명시적으로 갱신할 때 생성됩니다. DB 이벤트가 있으면 artifactSource=db_raw_events가 summary에 기록되고, 과거 run처럼 DB 이벤트가 없을 때만 기존 normalized artifact fallback을 사용합니다.

6단계: 세션 히스토리는 fingerprint로 관리한다

run-summary.json은 export history를 관리하기 위해 source metadata를 보존합니다.

artifactSource: db_raw_events
sourceAvailability.dbRawEventCount
sourceAvailability.dbRawMaxEventId
currentFingerprint
exportedFingerprint
exportStatus
dirty

새 raw event가 SQLite에 더 쌓이면 count/max id가 달라집니다. 이 값이 OTel artifact가 현재 DB 상태와 같은지 판단하는 기준입니다.

7단계: Trajectory도 명시 버튼으로 생성한다

handleTrajectoryGenerate()는 selected run의 projection input을 읽고 trajectory-evaluation.json을 씁니다.

const input = await readProjectionInput(projectPath, runId);
const trajectory = buildTrajectoryEvaluation(input);
await writeProjectionArtifact(projectPath, runId, "trajectory-evaluation.json", trajectory);

Trajectory의 judge.status는 처음에 not_run이어야 합니다. 실제 점수는 Claude judge, Opik, human judge가 나중에 명시적으로 확정합니다.

8단계: UI는 Book과 Evidence를 분리한다

현재 Desktop UI에서 상단 메뉴의 Book은 reader surface를 열고, Evidence는 runtime evidence surface를 엽니다.

surface파일역할
book-readerdesktop/src/components/v2/ViewRouter.tsx, BookVisualizationPage책 장과 Try Chat 진입점
runtime-evidencedesktop/src/components/v2/ViewRouter.tsx, BookVisualizationPageselected run의 artifact coverage, OTel, trajectory, Course Evidence
aux hostdesktop/src/components/v2/AuxSurfaceHost.tsxChat companion drawer/canvas로 Book/Evidence를 붙인다
chat menudesktop/src/components/v2/pages/TerminalPage.tsxSwarm/Team/History 옆에서 Book/Evidence 진입

파일 이름은 아직 BookVisualizationPage.tsx지만, UX 책임은 "책 시각화"가 아니라 Book Reader와 Runtime Evidence surface로 분리되어 있습니다.

9단계: Evidence surface가 보여 주는 것

Evidence surface는 selected run에 대해 다음을 보여 줍니다.

  • event count
  • artifact source
  • SQLite raw event count
  • raw archive 존재 여부
  • normalized events 존재 여부
  • OTel trace 존재 여부
  • trajectory 생성 여부
  • prompt score 저장 여부
  • SDK pipeline monitor
  • evidence chain
  • Course Evidence review
  • Trajectory artifact review

이 화면의 핵심은 "프롬프트 키워드가 아니라 artifact 경로와 source event id로 말한다"는 점입니다.

강좌 체크포인트

  1. SQLite raw_events와 JSONL export artifact를 분리한 이유는 무엇인가?
  2. Chat 실행과 Evidence refresh를 분리한 이유는 무엇인가?
  3. normalized-events.json이 Claude 평가와 Opik payload의 공통 입력이 되는 이유는 무엇인가?
  4. 파일 이름이 BookVisualizationPage.tsx여도 UX 책임을 문서에서 정확히 분리해야 하는 이유는 무엇인가?
